# AGENTS.md

This file is the canonical entrypoint for AI agents (Claude Code, Cursor, Windsurf, Antigravity, Gemini, etc.) working
in this repository. It defines how agents should behave, where context lives, and which rules override tool defaults.

If you are an **AI agent**: read this file first, then follow the pointers below to the project's own documents. If you
are a **human maintainer**: edit it here. Do not fork it across tool-specific directories.

> **This file is intentionally project-agnostic.** It does not describe this project's tech stack, build commands, or
> coding conventions. Look in the documents it points to for that.

## Workspace policy

This repository uses `.agents/` as the canonical, tool-neutral workspace for all agent artifacts.

- Use `AGENTS.md` (this file) as the project entrypoint. Do not duplicate it.
- Put reusable task instructions under `.agents/skills/<skill-name>/SKILL.md`.
- Put assets shared across skills (templates, snippets) under `.agents/skills/_shared/`. A `_`-prefixed directory there is not a skill (no `SKILL.md`) and tools (harnesses, `r3-artifact-audit`) skip it.
- Put custom agent definitions under `.agents/agents/<name>.md`. Use YAML frontmatter to set model, tools, and system prompt.
- Put project-agnostic coding rules under `.agents/rules/<rule-name>.md`. Rules apply automatically — agents must read the relevant rule before generating code in that domain.
- The SDD conventions (`sdd-schema`, `sdd-spec-format`, `sdd-domain-format`) are **canonical under `.agents/skills/_shared/`** and the `r3-sdd-*` skills **read them explicitly**, so they work on any tool. They are also symlinked into `.agents/rules/` (`sdd-*.md → ../skills/_shared/sdd-*.md`) for harnesses that scan that folder; the source of truth stays in `_shared/`.
- Put Warp terminal automation workflows under `.agents/workflows/<name>.yaml` (single `command` string plus `arguments`; Warp discovers them through the `.warp/workflows` symlink).
- Notes are distributed across two levels:
    - **Workspace notes** — `.agents/notes/` — cross-cutting concerns, shared patterns, and decisions that span projects. Entry point: `.agents/notes/INDEX.md`.
    - **Project notes** — `<project>/.agents/notes/` — architecture, milestones, and design decisions specific to one project. Entry point: `<project>/.agents/notes/INDEX.md`.
- Notes use Markdown with WikiLinks (`[[note-name]]`) to interconnect across levels. The WikiLink MCP resolves links across the full vault (workspace root as vault root).
- Organization within notes is freeform — no mandatory subdirectory structure. Use WikiLinks for navigation instead of folder hierarchy.
- Temporary agent notes go inline in the relevant note file, not in a separate directory.
- Do not create tool-specific instruction directories in this repository (`.claude/`, `.cursor/`, `.windsurf/`,
  `.antigravity/`, `.gemini/`, or similar). **Exception:** the following tool-specific entries are generated by
  `./setup.ps1 init`, gitignored, and must never contain duplicated content — only symlinks back into `.agents/`:
    - `.claude/skills → ../.agents/skills`
    - `.claude/rules → ../.agents/rules`
    - `.claude/agents → ../.agents/agents`
    - `.warp/workflows → ../.agents/workflows`
- `.mcp.json` at the workspace root is the canonical MCP server registry consumed by all harnesses (Claude Code,
  Warp, Gemini CLI). It is committed to the repo. Projects get their own `.mcp.json` scaffold via `projects.ps1 wire`.

## Available skills

Skills live under `.agents/skills/<skill-name>/SKILL.md`. Invoke them by name. When in doubt, read the skill file
before using it.

### Sources — reference library management

| Skill                | Purpose                                                                      |
| -------------------- | ---------------------------------------------------------------------------- |
| `r3-sources-link`    | Clone an upstream repo into the local sources store and register it          |
| `r3-sources-sync`    | Update one or all registered sources to their latest remote state            |
| `r3-sources-restore` | Re-clone missing stores and re-create missing symlinks from the registry     |
| `r3-sources-read`    | Explore and map a specific cloned source on demand                           |
| `r3-sources-learn`   | Pre-implementation methodology: evaluate sources, confirm, synthesize a plan |

### Git — version control workflows

| Skill            | Purpose                                                             |
| ---------------- | ------------------------------------------------------------------- |
| `r3-git-commit`  | Group pending changes into logical conventional commits             |
| `r3-git-release` | Analyze commits, propose semver bump, create release commit and tag |

### Project — workspace project management

| Skill               | Purpose                                                                      |
| ------------------- | ---------------------------------------------------------------------------- |
| `r3-project-add`    | Clone a project repo into the workspace and wire up agent config inheritance |
| `r3-project-list`   | List all active projects in the workspace                                    |
| `r3-project-remove` | Remove a project from the workspace with safety checks                       |

### Artifact — cross-tool artifact management

| Skill                 | Purpose                                                                                                      |
| --------------------- | ------------------------------------------------------------------------------------------------------------ |
| `r3-artifact-create`  | Create a skill, rule, agent definition, Warp workflow, or MCP server with correct structure and side effects |
| `r3-artifact-audit`   | Scan all artifacts for cross-tool consistency issues (missing files, wrong extensions, unregistered entries) |
| `r3-artifact-improve` | Audit and refactor a single skill against the canonical format (audit-first, applies on approval)            |
| `r3-artifact-remove`  | Remove an artifact and clean up all side effects (AGENTS.md entries, JSON keys, etc.)                        |

### SDD — spec-driven development

Engine-free spec-driven development. A living source contract (`.covenant/specs/`, edited directly — git is the
history) + numbered per-change records under `.covenant/changes/`; the agent performs all logic (scaffold, status,
close, verify) — see the conventions `sdd-schema`, `sdd-spec-format`, and `sdd-domain-format` (in `.agents/skills/_shared/`).

| Skill                 | Purpose                                                                                           |
| --------------------- | ------------------------------------------------------------------------------------------------- |
| `r3-sdd-init`         | Initialize the `.covenant/` structure (specs, changes, constitution, context-map, explorations)   |
| `r3-sdd-propose`      | Propose a change and generate all planning artifacts (proposal, specs, design, tasks) in one pass |
| `r3-sdd-scaffold`     | Scaffold a change and show the first artifact to fill, then stop (step-by-step)                   |
| `r3-sdd-continue`     | Create the next artifact for an in-progress change (one at a time)                                |
| `r3-sdd-fast-forward` | Fast-forward — generate every remaining planning artifact until apply-ready                       |
| `r3-sdd-explore`      | Explore mode — investigate and clarify before committing to a change (never implements code)      |
| `r3-sdd-analyze`      | Pre-implementation coverage + consistency check (read-only; requirement↔task by `REQ-###`)        |
| `r3-sdd-checklist`    | Requirement-quality checklist for a concern — "unit tests for English" (optional)                 |
| `r3-sdd-apply`        | Implement a change's tasks, checking them off in `tasks.md`                                       |
| `r3-sdd-close`        | Close a change: fold it into the living source specs by direct edit, verified via `git diff`      |
| `r3-sdd-verify`       | Verify an implementation matches the change's artifacts (read-only self-check)                    |
| `r3-sdd-reconcile`    | Brownfield gap-close — assess code vs spec/tasks, append gap-closing tasks (optional)             |
| `r3-sdd-onboard`      | Guided tutorial through one full SDD cycle on the user's codebase                                 |

## Source verification policy

Memory is not a source. Never assume function signatures, APIs, or behaviors. Verify against actual source files or
official documentation before writing code.

- **Truth of the disk:** Rely strictly on local filesystem inspection. Use directory listing and file-reading tools to
  traverse symlinks and analyze the physical source code.
- **Source code of third-party libraries:** When you need to read or understand the implementation of a third-party
  library (internals, signatures, behavior), always look in `.agents/sources/<name>/` first. If the source is not
  registered there, report it and ask the user to run `/r3-sources-link <url>` — do not fall back to the internet for
  this purpose.
- If a local source appears unpopulated or broken, halt and ask the user to re-run `/r3-sources-link` or `/r3-sources-sync`.
- **General research (web allowed):** For everything else — evaluating alternatives, checking concepts, reading official
  API docs, comparing libraries, troubleshooting — web search is allowed and preferred over answering from memory.

## Local sources infrastructure

Reference sources are managed via `.agents/scripts/sources.ps1`:

- **Store:** actual clones live in `~/.r3/sources/<name>/` (global, shared across projects)
- **Links:** symlinks at `.agents/sources/<name>/` point to the store
- **Registry:** `.agents/registry.json` tracks all registered sources

```bash
sources.ps1 list                                      # list registered sources
sources.ps1 link <url> [name] [--branch B] [--full]   # clone and register
sources.ps1 sync [name]                               # update one or all
sources.ps1 restore [name]                            # re-clone missing stores, re-create missing symlinks
sources.ps1 remove <name> [--purge]                   # remove symlink (--purge deletes store)
```

Skills that read sources operate through `.agents/sources/<name>/` — the symlink is transparent.

## Communication and review style

Use direct, no-nonsense communication.

- **Maintainer language:** Spanish when communicating with the maintainer.
- **Code language:** English for code, identifiers, technical terms, documentation files, and comments unless explicitly
  requested otherwise.
- Do not sugarcoat technical problems.
- Be direct and specific. Do not soften technical problems or invent unknown APIs, signatures, or behavior.

### Review structure

When reviewing code or making technical decisions, use this structure:

1. **Verdict** — Immediate gut reaction in one line.
2. **Breakdown** — What is wrong or right, with surgical precision.
3. **Consequences** — Why it matters and what breaks if not fixed.
4. **Solution** — What to do instead, with concrete code when applicable.

### Closing-task rules

- When done, confirm in one line what was completed.
- If there are pending user-owned tasks, mention them before stopping. Do not silently ignore them.

### Calibration rules

- If something is correct, a simple "correct" is enough. Do not embellish the obvious.
- If something is wrong, say it without filter. Be specific about what is wrong and why.
- If you do not know something, say so.
- Memory is not a source.
- Prioritize what matters. Do not waste time on cosmetic details when there are structural problems.

## Multi-LLM execution matrix

Adapt your reasoning engine dynamically based on the specific model the user selects for the current execution thread:

- **Claude (Sonnet / Opus, thinking mode):** Master software artisan. Focus on precise, production-grade source code,
  target language idioms, clean type handling, and structured file-system persistence inside the worktree.
- **Gemini Pro (Ultra / Advanced):** Global context ingestion engine. Use the large context window to analyze large
  codebases, trace dependency trees, and map cross-cutting architectural discrepancies without truncation.
- **OpenAI reasoning models (o1 / o3 / GPT-4o):** Deep algorithmic logician. Focus on structured logic queries to solve
  concurrency patterns, race conditions, or performance bottlenecks.

## Token budgeting and checkpoints

- **Review-driven checkpoints:** Before launching any parallel file-system analysis task or sub-process, present a
  2-line declarative statement outlining _what_ you will scan on the disk and _why_.
- **The switch pause:** Pause immediately after the declarative statement to allow the user to manually switch the
  active LLM model in the IDE interface depending on the nature of the task (analysis vs. coding).
- **Pruned file diffs:** When applying modifications, generate code blocks targeting only the exact delta required.
  Avoid full file rewrites to conserve the active credit pool.

## Worktree and branch naming

When creating a new worktree or Git branch, use the following lowercase kebab-case format:

```
feature/<identifier>-<descriptive-name>
```

- `<identifier>`: 2-digit sequential number, milestone prefix, or task code (e.g., `01`, `02`, `core`).
- `<descriptive-name>`: 1–2 English keywords summarizing the scope, separated by hyphens.
- Examples: `feature/01-primitives`, `feature/02-secrets`, `feature/03-transport`.
- Never use generic, random, or auto-generated IDE names.

---

## Security and safety

- Never commit secrets, credentials, tokens, private keys, or local machine paths that should not be shared.
- Never log secret values. Logging that includes config blocks must redact `secret.*` references before emitting.
- Ask before destructive operations on data, branches, history, or external source checkouts.
